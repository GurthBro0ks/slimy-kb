# Question
What are the key architectural patterns and principles that govern how SlimyAI agents operate across NUC1 and NUC2? Include the harness structure, cross-NUC communication, and any known failure modes.

# Answer
SlimyAI operations are governed by a contract-first harness plus strict host-boundary architecture.

## 1) Harness structure (control plane for all agent work)
- The primary harness is a session contract: every run starts with shared context loading, follows repo-local AGENTS rules, and closes with verification + commit discipline. (Citation: [Agent Session Contract](../wiki/concepts/agent-session-contract.md), [Session Closeout Pattern](../wiki/patterns/session-closeout-pattern.md))
- Three ledgers are treated as operational truth across sessions: `claude-progress.md` (history), `feature_list.json` (pass/fail state), and `server-state.md` (infra ownership/health). This is the continuity layer across agents and hosts. (Citation: [Source of Truth Ledgers](../wiki/concepts/source-of-truth-ledgers.md))
- Completion is defined by a truth gate (lint/tests/runtime checks + explicit pass/fail recording), not by implementation alone. (Citation: [Truth Gate](../wiki/concepts/truth-gate.md))

## 2) Cross-NUC architecture and communication principles
- Placement is explicit by host: NUC1 runs infra dependencies (MySQL, Caddy, chat, agent-loop), while NUC2 runs web/control surfaces (Next.js web, mission-control, legacy PostgreSQL). (Citation: [NUC Topology and Services](../wiki/architecture/nuc-topology-and-services.md))
- Cross-NUC data access is treated as a controlled boundary: cross-host DB access requires tunnel validation and explicit MySQL grants, rather than implicit LAN trust. (Citation: [NUC Topology and Services](../wiki/architecture/nuc-topology-and-services.md), [Q1 2026 Operational Fixes](../wiki/troubleshooting/q1-2026-operational-fixes.md))
- Coordination interfaces are service-based (task/agent/comms/webhook endpoints in Mission Control), not ad hoc process coupling. (Citation: [Mission Control](../wiki/projects/mission-control.md))
- Architecture includes negative constraints: retired services/components (admin-api/admin-ui/Discord OAuth login path) are intentionally dead and must not be resurrected. (Citation: [Auth and Retired Services](../wiki/architecture/auth-and-retired-services.md))

## 3) Known failure modes (already observed)
- Tunnel MySQL access denied from bridge IP due missing host grants on NUC1. (Citation: [Q1 2026 Operational Fixes](../wiki/troubleshooting/q1-2026-operational-fixes.md))
- Environment DB target drift (wrong schema in web env) causing route/database failures. (Citation: [Q1 2026 Operational Fixes](../wiki/troubleshooting/q1-2026-operational-fixes.md))
- Bot OpenAI 401s from stale/invalid API key in runtime env. (Citation: [Q1 2026 Operational Fixes](../wiki/troubleshooting/q1-2026-operational-fixes.md))
- Contract-visible SSR content missing because content was rendered only client-side. (Citation: [Q1 2026 Operational Fixes](../wiki/troubleshooting/q1-2026-operational-fixes.md))

## Confidence Level
Medium-high. The wiki clearly defines topology, guardrails, and recurrent operational failures, but some cross-NUC communication details remain implicit.

## Gaps Identified
- The wiki is strong on DB tunnel/grant cross-host communication, but lighter on a full map of all cross-NUC channels (for example: repo sync flow, mailbox transport, and service-to-service API paths).
- Runtime ownership details (for example, authoritative supervisors per service) are only partially captured in architecture pages.

## Wiki Gaps
- Missing dedicated architecture article: "Cross-NUC Communication Matrix" (protocol, direction, auth, failure signatures, owner).
- Missing dedicated architecture article: "Harness Runtime Topology" (which harness functions are host-local vs shared, and where agent-loop/mailbox components sit).
